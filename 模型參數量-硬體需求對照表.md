# 模型參數量與硬體需求對照表

日期：2026-04-27

這份表是給本專案做 **本地推理 / oMLX 架構規劃** 用的粗估表，不是官方部署規格。

重點假設：

- 主要看 **inference / serving**，不是訓練。
- 預設本地可用的主線是 **INT4 / Q4 權重量化**。
- Q8 / BF16 權重大小用來估算上限，不代表一定有足夠算力或頻寬。
- KV cache 會隨 `context length`、`batch / concurrency`、attention 架構而大幅變化。
- Apple Silicon 看的是 **UMA 總記憶體**；NVIDIA 看的是 **GPU VRAM**。
- MoE 模型要分開看 **總參數** 和 **active parameters**，不能只用總參數判斷。

## 2. 權重大小粗估

| 參數量 | Q4 / INT4 | Q8 / INT8 | BF16 / FP16 |
| ---: | ---: | ---: | ---: |
| 3B | 1.8GB | 3GB | 6GB |
| 7B | 4GB - 5GB | 7GB - 8GB | 14GB |
| 14B | 8GB - 10GB | 14GB - 16GB | 28GB |
| 32B | 18GB - 24GB | 32GB - 38GB | 64GB |
| 70B | 40GB - 48GB | 75GB - 85GB | 140GB |
| 120B | 70GB - 85GB | 125GB - 140GB | 240GB |
| 200B | 120GB - 140GB | 210GB - 230GB | 400GB |
| 405B | 240GB - 280GB | 430GB - 470GB | 810GB |
| 1T dense | 550GB - 700GB | 1TB+ | 2TB |

估算規則：

```text
BF16 / FP16 ~= parameters * 2 bytes
Q8 ~= parameters * 1.0 - 1.15 bytes
Q4 ~= parameters * 0.55 - 0.70 bytes
```

Q4 不是剛好 `0.5 byte / param`，因為通常還有：

- scale / zero point
- group metadata
- tensor alignment
- tokenizer / embedding / norm / lm_head 等非同等量化部分

## 3. KV cache 對硬體需求的影響

權重只決定「模型能不能放進去」，KV cache 決定：

- context 能開多長
- 同時能服務幾個 session
- decode 時 memory bandwidth 會不會爆

KV 大小沒有單一答案，取決於：

- layers
- KV heads
- head dimension
- precision
- context length
- batch / concurrency
- attention 架構：MHA / GQA / MQA / MLA

一般估算：

```text
KV bytes ~= layers * 2(K+V) * kv_heads * head_dim * bytes_per_value * tokens * batch
```

對 MLA 模型，例如 Kimi K2.6 / DeepSeekV3 類，KV 估算方式會不同，因為它保存的是 latent KV，而不是完整每 head K/V。

## 8. Dense 與 MoE 要分開看

Dense 模型：

```text
每個 token 幾乎都會用到全部權重。
硬體需求主要看 total parameters。
```

MoE 模型：

```text
每個 token 只啟用部分 experts。
硬體需求要同時看：
- total parameters
- active parameters
- expert cache hit rate
- router 導致的 unique experts / layer
- SSD / CPU / GPU 間搬運能力
```

所以：

- `70B dense` 和 `1T MoE / 32B active` 不能只用 active params 比。
- MoE 的計算量可能接近 30B 級，但權重儲存與 expert 調度是 1T 級問題。
- oMLX 的核心價值就在於把 1T MoE 變成可分層調度的 working set。

## 9. Kimi K2.6 類模型的特殊判斷

Kimi K2.6 不是普通大模型，它的關鍵特性是：

- 1T total params
- 32B active params
- native INT4
- 61 layers
- 384 experts
- 8 experts per token
- MLA
- 256K context
- HF 權重檔約 595GB

這代表：

| 子系統 | 壓力 |
| --- | --- |
| 權重容量 | 極高，無法完整常駐 256GB UMA |
| Active expert cache | 極高，是能否 decode 的關鍵 |
| SSD expert store | 必須，不是選配 |
| KV cache | 因 MLA 比標準 full KV 小很多 |
| 多用戶 | 受 expert route entropy 與 KV 共同限制 |
| 1M context | 非官方原生能力，屬於額外研究 |

對 K2.6 類模型，硬體需求不應寫成「需要多少 RAM 就能跑」，而應寫成：

```text
256GB UMA + 高速 SSD + expert paging + active expert cache + route-aware scheduler
```

如果沒有 expert paging，595GB 權重檔直接超過單機 UMA 預算。

## 11. 最終結論

小模型看的是：

> **權重能不能放進 RAM / VRAM。**

中大型 dense 模型看的是：

> **權重 + KV + context 是否還有 headroom。**

超大 MoE 模型看的是：

> **總權重能否被拆成 active working set，並由 SSD / UMA / GPU 穩定餵給 runtime。**

因此對本專案來說，硬體表的真正結論是：

> **7B - 70B 是一般本地推理問題；100B - 122B 是大記憶體本地工作站問題；Kimi K2.6 / 1T MoE 則是 oMLX 的 memory hierarchy 系統問題。**

## 12. 套用 oMLX 全技術棧後的估算方式

前面 1-11 節是「一般本地推理」的硬體需求。  
如果把 [oMLX-系統架構模組圖.md](./oMLX-系統架構模組圖.md) 內的技術全部估進去，硬體需求要改成看下面這個總式：

```text
Required UMA
~= resident shared weights
 + active expert cache
 + expert staging / prefetch buffers
 + hot / warm KV working set
 + KV metadata / page tables
 + scheduler / session / runtime buffers
 + Metal temporary buffers
 + safety headroom
```

對 dense model：

```text
active expert cache = 0
SSD cold expert store = optional / 不建議作為主線
主要收益來自 KV 壓縮、paged KV、prefix cache、Metal fused kernels
```

對 MoE model：

```text
full weight files 放 SSD
UMA 只保留 shared trunk + router + hot experts + staging
主要成敗取決於 active expert cache hit rate 與 SSD prefetch 是否能跟上 decode
```

重要提醒：

**下列技術不能全部線性相乘。**  
例如 SnapKV、MiniCache、QJL、Quest 都會讓 KV 變小或少讀，但它們作用在不同維度，疊加時會遇到品質、metadata、kernel overhead、I/O overlap 的上限。

## 13. oMLX 技術模組對容量與硬體需求的影響

## 14. oMLX 技術疊加後的 KV 估算

### 14.1 KV 有效倍率

以 BF16 / FP16 KV 作為 1x 基準：

| 技術組合 | 適用場景 | 保守有效 KV 倍率 | 中性有效 KV 倍率 | 激進有效 KV 倍率 |
| --- | --- | ---: | ---: | ---: |
| Paged KV + prefix cache | 多 session / 重複 prefix | 1.1x | 1.3x - 1.8x | 2x+ |
| SnapKV | 長 prompt / prefill 後減量 | 1.2x - 2x | 2x - 4x | 4x+ |
| MiniCache / xKV | layer-wise 冗餘高 | 1.2x | 1.5x - 2x | 2x+ |
| PolarQuant / QJL | KV precision 壓縮 | 2x - 3x | 3x - 5x | 5x - 8x |
| AdaptCache / EvicPress | hot/warm/cold placement | 1.5x | 2x - 4x | 4x+ |
| Quest + SSD KV prefetch | cold KV sparse read | 2x read reduction | 4x - 8x read reduction | 10x+ read reduction |

建議在硬體估算時使用下面的總倍率，而不是把上表全部相乘：

| 場景 | 可採用的總 KV 有效倍率 |
| --- | ---: |
| 短 context / 單用戶 | 1x - 1.5x |
| 32K - 128K / 單用戶 | 2x - 4x |
| 128K - 256K / 少量用戶 | 3x - 6x |
| 256K+ / 多用戶 | 4x - 10x |
| 1M context 研究 | 8x - 20x，但品質風險高 |

### 14.2 KV 工作集估算表

以下以「原始 BF16 KV」為 100%，估算 oMLX KV stack 後熱工作集可能降到多少。

| 模型級距 | 原始 KV 壓力 | oMLX 保守 | oMLX 中性 | oMLX 激進 |
| --- | --- | ---: | ---: | ---: |
| 7B - 14B | 中 | 70% - 100% | 35% - 60% | 15% - 35% |
| 27B - 34B | 中高 | 60% - 90% | 25% - 50% | 10% - 25% |
| 70B | 高 | 50% - 80% | 20% - 40% | 8% - 20% |
| 100B - 122B dense | 很高 | 45% - 75% | 15% - 35% | 5% - 15% |
| MLA MoE / K2.6 類 | 單 session 中，多 session 高 | 60% - 90% | 25% - 50% | 10% - 25% |

K2.6 類 MLA 模型要特別看待：

- MLA 已經先把 KV 從 full KV 壓小。
- 所以 PolarQuant / QJL 的相對收益仍在，但不應再用 full-KV 的數百 GB 作起點。
- 對 K2.6，KV stack 的最大價值是 **多 session 與 256K+ context**，不是單 session 能否放下。

## 15. oMLX 技術疊加後的 MoE 權重估算

MoE 權重估算不能只看 total params，還要看：

```text
resident trunk/router/shared weights
+ active expert cache
+ staging buffers
+ cold expert files on SSD
+ route locality
```

### 15.1 1T MoE 權重分層粗估

以 Kimi K2.6 類 `1T total / 30B-40B active / native INT4 / 權重檔約 500GB-700GB` 為例：

| 區塊 | 256GB UMA PoC 估算 | 512GB UMA 較舒適估算 | 備註 |
| --- | ---: | ---: | --- |
| Resident trunk / router / shared path | 30GB - 60GB | 40GB - 90GB | 實際需依非 expert 權重比例校正 |
| Active expert cache | 50GB - 90GB | 120GB - 220GB | 越大越能降低 SSD miss |
| Expert staging / double buffer | 16GB - 32GB | 32GB - 64GB | prefetch pipeline 必備 |
| Hot / warm KV | 20GB - 60GB | 60GB - 140GB | K2.6 MLA 單 session 較小，多 session 才吃緊 |
| Scheduler / metadata / runtime | 8GB - 16GB | 16GB - 32GB | 多用戶下上升 |
| Metal temp / safety headroom | 16GB - 24GB | 32GB+ | 避免峰值爆掉 |
| SSD cold experts | 500GB - 700GB+ | 500GB - 700GB+ | 建議至少 1TB - 2TB 可用空間 |
| SSD cold KV / temp | 100GB - 500GB | 500GB - 2TB | 取決於 context 與 session 數 |

### 15.2 Expert cache hit rate 對 SSD 頻寬的壓力

以 K2.6 類模型粗估，單 token selected route 跨 MoE layers 的 expert 權重約 `10GB - 12GB` 級別。  
如果 active expert cache 沒命中，就不可能每 token 都從 SSD 讀這麼多資料。

| Active expert cache hit rate | cold expert read / token 粗估 | 對 decode 的意義 |
| ---: | ---: | --- |
| 90% | 約 1.0GB - 1.2GB / token | 只適合很慢 decode，容易被 SSD I/O 卡住 |
| 95% | 約 0.5GB - 0.6GB / token | 仍然很重，tok/s 一高就危險 |
| 98% | 約 200MB - 250MB / token | 可做低併發 PoC |
| 99% | 約 100MB - 120MB / token | 開始接近可用 |
| 99.5%+ | 約 50MB - 60MB / token | 多 token/s 才比較有希望 |

因此 1T MoE 的 oMLX 硬體需求不是單純：

```text
RAM >= active parameters size
```

而是：

```text
UMA 足夠大，讓 expert cache hit rate 高到 SSD read / token 可接受。
```

## 17. 256GB Apple Silicon 的 oMLX 分帳建議

若以 256GB Mac、有效可調度約 192GB 估算，套用全部 oMLX 技術後可分成三種 profile。

### 17.1 Dense 長 context profile

適合 70B - 122B dense。

| 區塊 | 建議預算 |
| --- | ---: |
| Q4 / mixed precision weights | 45GB - 85GB |
| KV hot / warm working set | 70GB - 95GB |
| KV metadata / Quest index | 1GB - 4GB |
| SSD KV prefetch buffers | 8GB - 16GB |
| Runtime / scheduler / Metal temp | 16GB - 24GB |
| Safety headroom | 12GB - 24GB |

這個 profile 的重點是：

- KV stack 是主角。
- SSD cold KV 可以延長 context。
- Dense 權重不能像 MoE expert 一樣大量冷放 SSD，否則每 token 都會被權重 I/O 卡死。

### 17.2 MoE model-scale-first profile

適合 K2.6 / DeepSeekV3 類 1T MoE PoC。

| 區塊 | 建議預算 |
| --- | ---: |
| Resident trunk / router / shared path | 30GB - 60GB |
| Active expert cache | 60GB - 90GB |
| Expert staging / prefetch | 24GB - 32GB |
| MLA KV hot / warm working set | 20GB - 50GB |
| KV metadata / page tables | 1GB - 4GB |
| Runtime / Metal temp | 16GB - 24GB |
| Safety headroom | 12GB - 24GB |

這個 profile 的重點是：

- 權重側優先，KV 先不要吃滿 95GB。
- K2.6 因 MLA，單 session 256K KV 約十幾 GB 級，不必一開始就把最大預算給 KV。
- active expert cache 太小會導致 SSD read / token 爆掉。

### 17.3 Multi-user serving profile

適合 7B - 70B，或 100B 以上低併發。

| 區塊 | 建議預算 |
| --- | ---: |
| Resident weights / active experts | 40GB - 90GB |
| KV hot cache | 50GB - 80GB |
| Warm / compressed KV | 20GB - 40GB |
| Prefix cache / metadata | 2GB - 8GB |
| Scheduler / sessions / request queues | 4GB - 12GB |
| Prefetch / staging buffers | 8GB - 24GB |
| Runtime headroom | 16GB - 24GB |

這個 profile 的重點是：

- continuous batching 和 prefix cache 提升吞吐，但會放大 KV 與 expert route 壓力。
- admission control 必須同時看 KV blocks、prefill 長度、active experts、SSD queue。

## 18. oMLX 技術全部納入後的最終硬體判斷

若只看一般本地推理：

> **模型越大，需要的 RAM / VRAM 越大。**

若把 oMLX 架構圖內技術都算進去，判斷會變成：

> **模型越大，越不能只問 RAM 多大，而要問 working set 能不能被分層、壓縮、預取、精準讀取。**

最終分界如下：

| 分界 | 判斷 |
| --- | --- |
| 7B - 14B | oMLX 技術主要提升長 context、多 session、serving 穩定度，不是讓模型「能不能放下」。 |
| 30B - 70B | KV stack 與 Metal fused path 開始明顯改變硬體需求，可讓 64GB - 128GB UMA 更實用。 |
| 100B - 122B dense | oMLX 可以降低 KV 與 bandwidth 壓力，但權重仍需大 UMA；192GB - 256GB 是合理區間。 |
| 200B+ dense | dense 權重本身太大，oMLX 只能延緩，不會魔法消除常駐權重需求。 |
| 1T MoE / K2.6 | oMLX 最有價值的區間；可把「完整常駐不可能」轉成「active working set + SSD expert store」問題。 |

一句話：

> **oMLX 全技術棧不能把小硬體變成無限大硬體，但能把硬體需求從「完整常駐全部資料」改寫成「只常駐當下真的會用到的權重與 KV」。這正是 1T MoE 類模型在 256GB / 512GB Apple Silicon 上有討論空間的原因。**
